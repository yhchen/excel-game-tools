# Testing System Design

## Context

`excel-game-tools` is a TypeScript CLI for reading Excel and CSV game configuration tables, validating and transforming them through the TypeDef DSL, then exporting JSON, JS, Lua, Go, proto2, proto3, and protobuf-net/C# outputs.

The repository already has a useful fixture smoke command:

- `npm test` runs `scripts/verify-fixtures.js`.
- `scripts/verify-fixtures.js` builds the project, runs the bundled `testcase/` CLI flow, checks representative generated outputs, parses selected JSON files, and checks proto syntax markers.

That command proves the broad happy path, but it is not yet a complete test system. It does not isolate specific TypeDef rules, it has limited negative coverage, and the existing `testcase/` fixtures mix public examples with regression coverage. Future changes to loaders, TypeDef parsing, validation rules, and exporters need more focused tests and richer CSV/XLSX cases.

The current working tree may contain unrelated exporter changes. This testing design should be implemented in a way that coexists with those changes and does not overwrite them.

## Goal

Create a formal test system using Node 22 built-in `node:test`, without adding a test framework dependency and without affecting `pkg` runtime packaging.

The test system should combine:

- focused unit tests for stable low-level rules,
- CLI integration tests against small dedicated fixtures,
- richer CSV and XLSX fixtures for edge cases,
- continued existing `testcase/` fixture verification for broad multi-export regression coverage.

## Non-Goals

- Do not introduce Vitest, Jest, ts-node, tsx, or another test framework.
- Do not require test dependencies in packaged standalone binaries.
- Do not replace the existing `testcase/` fixture flow.
- Do not rewrite public sample fixtures solely to support tests.
- Do not compare every generated JS/Lua/Proto/C# file byte-for-byte in the first version.
- Do not refactor loader, parser, TypeDef, or exporter internals as part of the test-system scaffolding unless a minimal testability seam is required and approved in the implementation plan.

## Chosen Approach

Use a mixed layered test strategy.

Dedicated tests live under `test/`, and test-only fixtures live under `test/fixtures/`. The existing `testcase/` directory remains the public sample and broad regression fixture set.

This approach keeps each fixture small and understandable, gives fast failure localization, and avoids turning `testcase/typeDef.js` or `src/config_tpl.json` into a large catch-all test harness.

## Test Directory Shape

Add this structure:

```text
test/
  integration/
    cli-fixtures.test.js
  unit/
    type-utils.test.js
    typedef-parser.test.js
    csv-loader.test.js
    export-common.test.js
  fixtures/
    positive/
      basic-types.csv
      arrays.csv
      groups.csv
      validators-and-defaults.csv
      objects-arrays-defaults.csv
      xlsx-merged-arrays.xlsx
      xlsx-comments-and-ignore.xlsx
    negative/
      required-empty.csv
      nonzero-zero.csv
      unique-duplicate.csv
      unique-default-duplicate.csv
      combined-required-unique-empty.csv
      combined-required-unique-duplicate.csv
      invalid-reference.csv
      invalid-enum.csv
      invalid-group.csv
      fixed-array-length.csv
      object-required-field.csv
      object-default-nonzero.csv
      json-invalid-syntax.csv
      json-required-field.csv
    configs/
      positive-*.json
      negative-*.json
    typeDefs/
      positive-typeDef.js
      negative-typeDef.js
      object-defaults-typeDef.js
    scripts/
      generate-xlsx-fixtures.js
  tmp/
```

`test/tmp/` is a generated output directory and should be ignored by git. Tests should clean the relevant case output directory before each CLI integration run.

All new XLSX fixtures should be generated by `test/fixtures/scripts/generate-xlsx-fixtures.js`, then committed as fixture files. This keeps binary fixture creation reproducible while still letting tests run without generating XLSX files every time.

## NPM Scripts

Keep `npm test` as the top-level validation entry point. Add scripts with clear responsibilities:

- `npm run test:unit`: build first, then run `node --test test/unit/*.test.js`.
- `npm run test:integration`: build first, then run `node --test test/integration/*.test.js`.
- `npm run verify:fixtures`: keep the current `scripts/verify-fixtures.js` flow.
- `npm test`: run unit tests, integration tests, and existing fixture verification.

Implementation can use a small `scripts/run-tests.js` if script composition becomes noisy, but it should not require a new dependency.

`npm run pkg && npm run verify:pkg` remains the packaging validation path. It is not required for every `npm test` run.

## Unit Test Scope

Unit tests should cover stable logic with high failure-localization value:

- `TypeUtils`: numeric extraction, valid and invalid numeric formats, and boundary string handling.
- `TypeDefParser`: basic type parsing, array splitting, default handling, and validators where the parser can be exercised without a full CLI process.
- CSV loader: quoted fields, empty fields, multiline fields, and sheet-name derivation.
- `export_common.ts`: only if the current exporter shared helper is intended to be stable common behavior; test focused helpers rather than every exporter string template.

Unit tests should import built files from `dist/` after `npm run build`. Test files should be plain JS to avoid adding TypeScript runtime tooling.

## Integration Test Scope

CLI integration tests should run the real compiled CLI in a child process:

```js
spawnSync(process.execPath, ['dist/index.js', '-c', configPath, '-t', typeDefPath], {
  cwd: rootDir,
  encoding: 'utf8'
});
```

Each positive case should assert:

- exit status is `0`,
- expected JSON output files exist,
- output JSON parses,
- key fields match expected parsed values.

Each negative case should assert:

- exit status is non-zero,
- stderr/stdout contains a stable error fragment such as `empty`, `Zero`, `duplicate`, `not found`, `Invalid`, `json`, or the relevant column/type name.

CLI tests should use child processes instead of importing and repeatedly calling `execute()`. This avoids cross-test pollution from global state such as `gCfg`, `ExportExcelDataMap`, `TypeDefParser.InitFlag`, and TypeDef initialization side effects.

## Fixture Strategy

New test fixtures should be small and scenario-oriented.

Positive fixtures:

- `basic-types.csv`: `int`, `uint`, `float`, `double`, `string`, `bool`, `date`, `tinydate`, `timestamp`, and `utctime` normal parsing, empty values, and defaults.
- `arrays.csv`: one-dimensional, two-dimensional, and three-dimensional arrays; fixed-length arrays; trailing separators; empty array cells.
- `groups.csv`: `$` group row behavior, `GroupMap`, multiple export filters, comment rows, comment columns, and empty rows.
- `validators-and-defaults.csv`: `DVAL`, `<!N>`, `<!0>`, `<!!>`, `<!N;!!>`, `<!0;!!>`, and `<!N;!0;!!>` passing cases.
- `objects-arrays-defaults.csv`: `TObject`, `TArray(TObject)`, and `TJson` fields with `DVAL` and validators.
- `xlsx-merged-arrays.xlsx`: merged-cell array columns, `colspan` headers, and Excel date cells.
- `xlsx-comments-and-ignore.xlsx`: ignored sheets, comment rows, comment columns, empty rows, and special sheet naming rules.

Negative fixtures should be one failure mode per file/config where practical. This prevents the first expected error from hiding later expected errors.

## Validator And Default Matrix

The first version must explicitly cover these TypeDef combinations.

Empty value behavior:

- plain `int`, `string`, `bool`, and date-like fields with empty cells,
- `int<!N>` and `string<!N>` with empty cells fail,
- `int.DVAL(5)` empty cell emits `5`,
- `int.DVAL(5)<!N>` empty cell passes and emits `5`,
- `string.DVAL('x')<!N>` empty cell passes and emits `x`,
- whitespace-only cells are tested and locked to current behavior.

Non-zero behavior:

- `int<!0>` with `0` fails,
- `float<!0>` with `0` and `0.0` fails,
- `int<!0>` with an empty cell follows current default behavior and is asserted,
- `int.DVAL(0)<!0>` with an empty cell fails because the effective value is `0`,
- `bool<!0>` is covered by a characterization test if the current TypeDef DSL accepts the combination; if the DSL rejects it during implementation, the test file must document that exclusion next to the non-zero validator cases.

Unique behavior:

- `int<!!>` duplicate numbers fail,
- `string<!!>` duplicate strings fail,
- `int<!!>` empty cell fails under current parser behavior,
- `int.DVAL(1)<!!>` multiple empty cells fail because they resolve to duplicate `1`,
- `string<!!>` case sensitivity is locked to current behavior,
- `int<!!>` numeric string and numeric value collision behavior is locked to current behavior.

Combined validators:

- `string<!N;!!>` empty value fails,
- `string<!N;!!>` duplicate value fails,
- `int<!0;!!>` zero fails,
- `int<!0;!!>` duplicate non-zero value fails,
- `int<!N;!0;!!>` covers empty, zero, duplicate, and valid unique non-zero values,
- validator order such as `<!N;!!>` and `<!!;!N>` is tested as behaviorally equivalent if the parser supports both forms.

Objects and JSON:

- `TObject({ count: int.DVAL(1) })` fills a missing field with `1`,
- `TObject({ count: int.DVAL(0)<!0> })` fails when the missing field resolves to `0`,
- `TObject({ name: string<!N> })` fails when the field is missing or empty,
- `TObject({ id: int<!!> })` behavior is tested and documented because uniqueness may depend on TypeDef instance state,
- `TArray(TObject({ count: int.DVAL(1) }))` fills each missing object field independently,
- `TArray(int, 3)` fails for too few or too many elements,
- `TArray(int<!0>)` fails when any element is `0`,
- `TJson({ key: int.DVAL(1), value: string<!N> })` fills missing `key` and fails missing `value`,
- invalid JSON syntax fails with a stable JSON-related error fragment.

Enums and references:

- enum legal numeric values pass,
- enum name-like values are covered by a characterization test if the current `def.TEnum` parser supports them; if not, the test file must document that enum fixtures intentionally use numeric values,
- enum illegal values fail,
- enum empty value with `<!N>` fails,
- enum `DVAL` passes and emits the default,
- `Sheet.Column` references pass for existing values,
- missing references fail,
- reference fields inside `TObject` fail with a stable reference-related error fragment.

## CSV And XLSX Edge Cases

CSV coverage should include:

- empty fields,
- quoted empty strings,
- quoted fields containing commas,
- multiline quoted fields,
- third-level array separators using newline content,
- CRLF normalization where relevant.

XLSX coverage should include:

- Excel date cells for `date`, `tinydate`, `timestamp`, and `utctime`,
- merged-cell array columns,
- merged header cells that produce `colspan`,
- comment rows,
- comment columns,
- ignored sheets,
- empty rows before and between data rows,
- illegal group names as a negative case.

## Error Handling Assertions

Tests should assert stable behavior, not incidental log formatting.

Positive tests should prefer exact JSON value assertions.

Negative tests should assert:

- non-zero exit code,
- a stable error fragment,
- the relevant fixture/config name when useful.

Negative tests should not depend on full stack traces, exact line wrapping, chalk color codes, or the order of multiple independent errors.

## Packaging Impact

The design avoids adding runtime or test framework dependencies. Tests use Node 22 built-in `node:test`, `node:assert`, `node:fs`, `node:path`, and `node:child_process`.

Because tests live outside the runtime CLI entry and no test framework is added to dependencies, `pkg` should not include test-only code in standalone binaries. Packaging remains validated by:

```bash
npm run pkg
npm run verify:pkg
```

## Acceptance Criteria

- `npm test` builds the project, runs unit tests, runs CLI integration tests, and runs existing fixture verification.
- No new test framework dependency is added.
- Tests are plain JS and import compiled `dist/` modules where unit coverage needs project code.
- CLI integration tests use child processes and isolated output directories under `test/tmp/`.
- `DVAL`, `<!N>`, `<!0>`, `<!!>`, `<!N;!!>`, `<!0;!!>`, and `<!N;!0;!!>` are covered.
- `TObject`, `TArray(TObject)`, and `TJson` cover field-level `DVAL` and validator combinations.
- CSV fixtures cover empty fields, quoted fields, multiline fields, and newline array separators.
- XLSX fixtures cover dates, merged cells, comments, ignored sheets, and invalid groups.
- Positive tests assert generated JSON values.
- Negative tests assert non-zero exit status and stable error fragments.
- Existing `scripts/verify-fixtures.js` still validates representative outputs under `testcase/exports/`.
- `npm run pkg && npm run verify:pkg` remains available and independent from normal `npm test`.

## Risks And Mitigations

Some TypeDef internals hold state across parser instances. CLI integration tests mitigate this by using a separate process per case or per test group.

Binary XLSX fixtures can be hard to review. The fixture generation script mitigates this by making their contents reproducible and reviewable as code.

Overly broad snapshot testing can make harmless formatting changes painful. The first version avoids full-file snapshots and focuses on exact JSON values plus stable generated-file smoke checks.

Negative tests can become brittle if they match complete logs. They should match short stable fragments tied to the semantic failure.

The fixture matrix can grow quickly. Keep each fixture focused, and add new fixtures only when they protect a specific rule, bug fix, or exported behavior.
